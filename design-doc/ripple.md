### 0.2 公共字段

若非特别说明，所有数据表都应包含一组标准公共字段，以记录元数据。配置表、关联表等特殊表除外。

详见：《后端项目开发规范（通用）》

```markdown
- `id` (主键): `UUID` 类型。
- `created_time` (创建时间)：`timestamptz` 类型。
- `updated_time` (更新时间)：`timestamptz` 类型。
- `deleted_time` (软删除时间)：可选，`timestamptz` 类型，`NULL` 表示未删除。
- `created_by` (创建者)：可选，`UUID` 类型，记录创建者 ID。`NULL` 表示由系统创建。
- `updated_by` (更新者)：可选，`UUID` 类型，记录更新者 ID。`NULL` 表示由系统更新。
- `deleted_by` (删除者)：可选，`UUID` 类型，记录删除者 ID。`NULL` 表示由系统删除。
- `status` (状态)：`smallint` 类型，数值详见字典初始化数据。
- `remark` (备注)：`text` 类型。
- `is_system_reserved` (系统保留)：`boolean` 类型，标识是否为系统保留数据，默认为 `false`，如果是，则不可删除。
```

#### 1.1.5 业务规则

##### 1.1.5.1 创建/修改逻辑

###### 后台管理面板的用户管理表单

- 创建/修改的时候，用户可以选择和修改所属角色。
- 前端的用户表单中，角色字段应为多选框，允许选择多个角色。
- 用户在创建时必须设置密码
- 在用户修改表单中，用户可以选择不修改密码，但是无论是否修改密码，都必须提供旧密码进行验证。

###### 用户修改自己资料（updateMy）

- 字段和后台用户管理表单相同，但不允许修改自己的角色。
- 用户在修改资料时，可以选择不修改密码，但是用户如果选择修改密码，则必须提供旧密码进行验证，否则报错，非重要信息（如昵称、邮箱等）则不需要验证旧密码。

##### 1.1.5.2 查询

- 普通用户只能查询和修改自己的资料，不能查询其他用户的信息，因此，用户管理的 `findOne` 接口的初始化的权限数据，不把 CRUD 的权限给普通用户。
- 管理员用户和超级管理员用户可以查询和修改所有用户的信息，这个权限通过菜单里的权限赋予实现。
- 上述权限不要通过代码逻辑实现。

##### 1.1.5.3 删除逻辑

- 删除用户前应检查是否有重要的关联数据，若有则阻止删除操作并返回错误信息。

##### 2.1.3.1 创建/修改逻辑

- 菜单的 `type` 字段用于区分菜单类型，例如目录、菜单项、按钮（又作接口）等。
- 创建/修改菜单时，前端应根据 `type` 字段动态调整表单项。例如，若 `type` 为目录，则不显示 `route` 字段。
- 菜单的 `permission_code` 字段必须唯一，不允许重复。
- 创建/修改菜单时，若指定了 `parent_id`，则必须验证该父级菜单是否存在，且不能是当前菜单本身或其子菜单，以防止循环引用。
- 菜单的排序字段 `sort` 应允许用户自定义，以控制菜单在前端的显示顺序。默认值为 `0`，数值越小，排序越靠前。
- 用户在创建/修改菜单时，可以选择该菜单所属的角色，实现权限控制。

- 当一个目录下面没有内容，或者只有按钮类型的菜单时，该目录在前端不显示。

- 权限赋予不得硬编码在代码中，必须通过系统菜单和角色来实现。

## 3. 登录鉴权

### 3.1 登录和注册

#### 3.1.1 DTO 和 VO

##### `LoginDTO`

```jsonc
{
  "name": "", // 用户名
  "password": "" // 密码
}
```

##### `LoginVO`

```jsonc
{
  "userId": "",
  "userName": "",
  "permissionCodes": ["string"], // 用户权限编码列表
  "systemRoles": [], // 用户角色编码列表
  "systemMenus": [], // 用户菜单编码列表，树表
  "accessToken": "", // JWT 令牌
  "refreshToken": "" // 刷新令牌
}
```

##### `RegisterDTO`

```jsonc
{
  "name": "", // 用户名
  "nickName": "", // 昵称
  "email": "", // 邮箱，注册时可选
  "phone": "", // 手机号，注册时可选
  "password": "" // 密码
}
```

##### `RefreshTokenDTO`

```jsonc
{
  "refreshToken": "" // 刷新令牌
}
```

##### `RefreshTokenVO`

```jsonc
{
  "newAccessToken": "", // 新的 JWT 令牌
  "refreshToken": "" // 新的刷新令牌
}
```

#### 3.1.2 接口

| 接口名称 | 方法 | 路径           | 描述                                 |
| -------- | ---- | -------------- | ------------------------------------ |
| 登录     | POST | /auth/login    | 用户登录，成功后返回 JWT             |
| 注册     | POST | /auth/register | 用户注册                             |
| 刷新令牌 | POST | /auth/refresh  | 刷新访问令牌，使用 refreshToken 刷新 |

#### 3.1.3 业务规则

- `guest` 保留用户禁止手动登录，前后端均需校验。
- `root` 和 `guest` 用户禁止通过注册接口创建，只能通过系统初始化来完成。
- 通过注册接口创建的用户，只分配普通用户 `user` 角色，管理员可以在后台管理端提升权限。
- `guest` 用户不需要 token，因此禁止访问 `/auth/refresh` 接口。

### 3.2 鉴权

#### 3.2.1 业务规则

- 凡是未登录的用户，自动赋予 `guest` 用户权限，拥有游客权限，登录、注册等接口应允许游客访问。
- RBAC 角色权限控制：每个用户可以拥有多个角色，每个角色对应一组权限。系统通过用户的角色来确定其访问权限。
- 遵照《后端开发规范》，每一个 API 方法都必须映射到唯一的权限点。
- 基于 `Casbin` 实现上述权限控制。
