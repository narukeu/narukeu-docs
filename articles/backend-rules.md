# 后端项目开发规范（通用）

本文档旨在为所有后端项目提供一套通用的开发规范，确保代码质量、可维护性和团队协作效率。本规范是语言和框架无关的，适用于任何后端技术栈。

目前后端项目首要使用 `PostgreSQL` 作为数据库，因此数据类型和 SQL 相关的规范均基于 `PostgreSQL`。

## 1. 架构与设计原则

### 1.1. 项目结构

应根据所使用的技术栈，采用社区推荐的项目结构。确保代码模块化、职责单一，便于维护和扩展。

- 至少要有 `Controller`、`Service`、`Repository/DAO` 三层。
- `Service` 层先定义接口再实现，对于 Java 和 JavaScript/TypeScript，`Service` 实现可以命名为 `{ServiceName}Impl`。
- 对于 Go 等没有类和接口概念的语言，可以通过定义接口类型和结构体来实现类似的分层效果。
- 应当以功能模块为单位划分代码文件和目录结构，确保每个模块职责单一，便于维护和扩展。
- 要有 `common` 文件夹，存放公共代码，比如，全局异常处理、中间件、工具函数等。

#### 1.1.1. 模块常量（路由前缀）

为提高路由管理的一致性并减少硬编码，推荐为每个包含路由定义的模块设置"模块路由前缀常量"。

- **定义位置**：在模块目录下的常量文件中（如 `constants` 文件）定义路由前缀常量。
- **层级组合**：子模块应基于父模块前缀进行拼接，形成层级化路由结构。
- **作用域限制**：该常量仅在模块内部使用，不应通过公共导出文件对外暴露。
- **版本集成**：与 API 版本管理结合使用，形如 `/v1/system/dict` 的完整路由路径。

### 1.2. 对象定义

根据不同的使用场景，后端项目中常见的数据对象包括以下几种：

- **数据模型 (Data Model)**：与数据库表结构一一对应的代码对象，在一些技术栈中也叫实体（Entity），在规范中统一称为 `Model`，无论技术栈如何。
  - 它用于数据操作。比如说，用户表，它的业务模型可以定义叫 `User`，业务模型原则上需要继承基础模型 `BaseModel`，在下面的章节会介绍。
  - 当然，如果一些表不需要公共字段，也可以不继承 `BaseModel`，比如说配置表、关联表等。
- **DTO (Data Transfer Object)**：用于在不同层之间（如客户端与服务端）传输数据的对象，通常作为 API 的输入参数。对于 DTO，可以这样命名：`CreateUserDTO`。由于 `CreateXXXDTO` 有很多字段是公共字段，因此可以定义 `BaseDTO` 供其他 DTO 继承，`BaseDTO` 的内容可以比照 `BaseModel`。
- **VO (View Object)**：用于服务端返回给客户端的视图层对象。如果其结构与 DTO 相同，可复用，命名方式为 `FindListVO`。

对于 Go 等语言，它是没有类的概念的，因此上述对象均使用结构体（Struct）来定义。
Go 等语言也因此没有继承的概念，对于 Go，可以使用结构体嵌套的方式来实现类似继承的效果。如果所使用的语言也不支持嵌套，可以使用组合的方式来实现类似效果。

## 2. 编码规范

### 2.1. 通用准则

- **代码自解释**：力求代码本身就能清晰地表达其意图，减少不必要的注释。
- **时间处理**：所有时间数据在存储、处理和传输时，统一使用 `UTC` 时间，以避免时区问题。
- **参数命名**：回调函数或方法签名中的参数名应具有明确的描述性，避免使用无意义的单字母变量（如 `i`, `v`, `e` 等）。

### 2.2. 注释规范

只在必要时添加注释。注释的核心是解释“为什么这么做”，而不是“这是什么”。应重点阐述代码背后的设计思想、业务背景或特殊考量。注释应使用中文。

## 3. API 设计规范

### 3.1. HTTP 方法与版本管理

- **HTTP 方法**：为简化客户端和网关的实现，API 仅使用 `GET` 和 `POST` 两种 HTTP 方法。不使用其他方法（如 `PUT`, `DELETE`, `PATCH` 等）。
  - `GET`：用于请求数据，参数通过 URL 查询字符串传递。
  - `POST`：用于提交数据（创建/更新）或执行需要复杂参数的查询操作。
- **API 版本管理**：所有 API 都应包含版本标识。推荐使用 URL 路径进行版本控制，例如 `/v1/system/user/findAll`。当发生不兼容的 API 变更时，应发布新的版本（如 v2），并为旧版本提供一段过渡期。

### 3.2. 基础接口命名

为了保持项目结构的一致性和可预测性，所有资源（Resource）的 CRUD 操作应遵循以下标准方法命名。控制器（Controller）和服务（Service）层的方法名必须保持一致。

- `findOne`：获取单个资源。
- `findList`：获取资源列表（分页）。
- `findAll`：获取所有资源列表（不分页）。
- `findTree`：获取树形结构的资源列表（可选），并分页，入参规范与 `findList` 相同。`findList` 和 `findAll` 只返回扁平结构。
- `findTreeAll`：获取所有资源的树形结构列表（可选），不分页，入参和 `findAll` 相同。
- `create`：创建单个资源。
- `createMany`：批量创建资源（可选）。
- `update`：更新单个资源。
- `delete`：永久删除单个资源。
- `deleteMany`：批量永久删除资源（可选）。
- `remove`：逻辑删除（软删除）单个资源。
- `removeMany`：批量逻辑删除资源（可选）。
- `restore`：恢复被逻辑删除的资源。
- `restoreMany`：批量恢复被逻辑删除的资源（可选）。
- `count`：获取资源总数（可选）。

### 3.3. 查询接口规范

#### 3.3.1. 复杂查询

- `findList`、`findAll`、`findTree`、`findTreeAll` 接口应使用 `POST` 方法，允许客户端通过请求体（Request Body）以 JSON 格式传递复杂的过滤和排序条件。

#### 3.3.2. 查询模式 (Query Mode)

- `findList`、`findAll`、`findTree`、`findTreeAll` 接口必须支持一个名为 `mode` 的 URL 查询参数，用于控制是否包含逻辑删除的数据。
- `mode` 参数的可选值如下：
  - `default`：默认模式，仅返回未被软删除的数据。
  - `withDeleted`：返回所有数据，包括已被软删除的数据。
  - `onlyDeleted`：仅返回已被软删除的数据。
- 查询列表时，默认按照 `created_time` 字段降序排序。

#### 3.3.3. 分页与排序参数

- 分页参数通过 `URL` 查询参数传递：
  - `currentPage`，表示当前页码，默认值为 `1`。
  - `pageSize`，表示每页记录数，默认值为 `20`。
- 对于 `findList` 等有筛选条件的接口，筛选参数应通过请求体（Request Body）以 JSON 格式传递。

由于默认已经按照创建时间倒序，因此一般情况下不需要传递排序参数。如果确实需要自定义排序，可以在 URL Query 中传递 `sortBy` 和 `sortOrder` 参数。

- `sortBy`：指定排序字段的名称。
- `sortOrder`：指定排序顺序，可选值为 `0`（降序）和 `1`（升序），默认值为 `0`。
- `sortOrder` 可以单独传递，如果只传递 `sortOrder`，则表示对默认的 `created_time` 字段进行排序。

`pageSize` 最大限制为 `150`，如果超过则报错。

分页具体实现，如果 ORM（如 `Mybatis-Plus`、`Gorm`）中有现成的分页功能，可以直接使用，否则自己封装统一的分页逻辑。

#### 3.3.4. 批量 ID 入参对象

在进行批量操作且入参仅为一组资源标识符（ID）时，建议统一采用“批量 ID 入参对象”，以提升接口的一致性与可读性。建议定义成 公共 `DTO`，叫做 `ByIdsDTO`。

- **结构约定（推荐）**
  ```json
  {
    "ids": [
      "a348f08c-8c9e-4bac-95bd-146750cfb064",
      "6f7d8b9c-1234-4abc-9def-0123456789ab"
    ]
  }
  ```
- **适用场景**：`deleteMany`、`removeMany`、`restoreMany`、`updateMany` 等仅凭一组 ID 即可完成的批量接口。
- **校验与约束**
  - `ids` 必须为非空数组；元素类型与主键类型一致（默认 `UUID`，推荐 `uuidv7`）。
  - 元素去重处理；禁止 `null`/空字符串；可设置最大长度上限（例如 100~500，视业务与性能评估）。
  - ID 必须为合法格式；对于非 UUID 的项目，应使用与项目主键类型相匹配的校验规则。
- **行为约定**
  - 任意 ID 不存在时，默认整体失败并给出明确的业务错误（也可按需求提供“跳过不存在”模式，需在接口文档中声明）。
  - 与权限控制联动：仅当调用者具备相应批量操作权限点时才允许执行。
  - 日志审计应记录请求 ID、操作人、影响的资源数量与样本 ID。

### 3.4. 响应结构

#### 3.4.1. 统一响应体

所有 API 响应都应遵循统一的 JSON 结构，以便客户端进行标准化处理。

```json
{
  "businessCode": 0, // 业务状态码，0 表示成功，非 0 表示具体错误类型，注意，这不是 HTTP 状态码！
  // HTTP 状态码，前端诸如 Axios 等库会自动处理该字段，后端无需在响应体中返回。
  "isSuccess": true, // 错误时为 false
  "message": "操作成功！", // 错误时返回错误信息
  "data": { "id": "a348f08c-8c9e-4bac-95bd-146750cfb064", "name": "示例数据" }, // 返回的数据对象，错误时候为 null
  "timestamp": "2025-11-05T07:44:01.608Z", // ISO 8601 格式的时间戳
  "errors": null // 可选，详细的错误信息列表，复杂业务错误时使用
}
```

- 响应还应包含 `X-Request-ID` 头，用于请求追踪。

#### 3.4.2. 分页响应结构

带分页的接口，`data` 字段的结构如下，并做成 VO 用于统一设置：

```json
{
  "total": 100, // 总记录数
  "currentPage": 1, // 当前页码
  "pageSize": 10, // 每页记录数
  "records": [] // 当前页的记录数组
}
```

#### 3.4.3. businessCode 状态码

- `0`：操作成功
- `1xxx`：系统级错误（如数据库连接失败、超时等）
- `2xxx`：认证/鉴权失败（包括 `withDeleted` 和 `onlyDeleted` 等权限不足导致的问题）
- `3xxx`：参数错误（如缺少必填参数、参数格式错误等）
- `4xxx`：非数据库的业务逻辑错误（包括 `withDeleted` 和 `onlyDeleted` 等权限不足的鉴权错误）
- `5xxx`：数据库相关错误
- `9xxx`：未知错误

### 3.5. 文件处理

- 应当支持流式下载和断点续传。

## 4. 数据持久化规范

### 4.1. 数据库（PostgreSQL）

#### 4.1.1. 表与字段

- **命名规范**：表名和列名必须采用下划线命名法（`snake_case`），例如 `user_profiles`。
- **数据类型**：
  - **时间类型**：统一使用带时区的 `timestamptz` 类型，时区必须为 `UTC`。
  - **精确数值**：对于金额、汇率等需要高精度计算的字段，必须使用 `decimal` 类型。
  - **大文本**：对于不确定长度的大段文本，应使用 `text` 类型。
- **公共字段**：所有数据表都应包含一组标准化的公共字段，以记录元数据。
  - `id` （主键）: 唯一标识符，数据库类型为 `UUID`。
  - `created_time` （创建时间）：记录创建时的时间戳，数据库类型为 `timestamptz`。
  - `updated_time` （更新时间）：记录最后一次更新时的时间戳，数据库类型为 `timestamptz`。
  - `deleted_time` （软删除时间）：可选，记录逻辑删除的时间戳，`NULL` 表示未删除，数据库类型为 `timestamptz`。
  - `created_by` （创建者）：可选，创建该记录的用户的 UUID，数据库类型为 `UUID`。允许为 NULL，用以表示该记录由系统自动创建或更新，而非某个特定用户。
  - `updated_by` （更新者）：可选，最后一次更新该记录的用户或系统标识，数据库类型为 `UUID`。允许为 NULL，用以表示该记录由系统自动创建或更新，而非某个特定用户。
  - `deleted_by` （删除者）：可选，逻辑删除该记录的用户的 UUID，数据库类型为 `UUID`。允许为 NULL，用以表示该记录由系统自动删除，而非某个特定用户。
  - `status` （状态）：记录当前的状态（通用版本），`0` 代表无效，`1` 代表有效，数据库类型为 `smallint`。
  - `remark`（备注）：用于存储额外信息的文本字段，数据库类型为 `text`。
- **主键**：主键 ID 应该使用 `uuidv7`，如果数据库支持，可以使用 `uuid_generate_v7()` 函数生成。如果不支持，则在应用层生成后再写入数据库。
- **模型定义**：如果定义了 `BaseModel`，应包含上述公共字段，其他数据模型继承该基础模型。这个模型里必须显式地定义这些字段，不使用 ORM 的隐式字段。
- **状态说明**：`status` 字段表示资源本身的业务状态，而非逻辑删除状态。逻辑删除应通过 `deleted_time` 字段来实现，两者职责分离。

#### 4.1.2. 关系 (Relation)

- **关系类型**：应根据业务需求选择合适的关系类型（如一对一、一对多、多对多），避免滥用多对多关系。
- **不用外键**：外键并不适合所有场景。在高并发写入场景下，外键可能导致性能瓶颈。对于关系，可以通过应用层逻辑来维护数据完整性，而非依赖数据库外键约束。
  - 由于不采用数据库外键约束，数据一致性的保障责任完全转移到应用层。开发人员必须通过以下方式确保数据完整性：
    - 在执行关联写入操作时，必须使用事务。
    - 在删除或更新关联数据时，必须先检查是否存在相关联的数据，防止出现孤立记录。
    - 定期运行数据完整性检查脚本，发现并修复潜在的数据不一致问题。

#### 4.1.3. 树形结构

- **实现方式**：为了性能考虑，尽可能用 SQL 语句实现树结构的查询和操作，避免在应用层进行递归处理。当然，如果你所使用的 `ORM` 有现成的树结构的支持（包括在树结构中筛选节点），也可以使用 `ORM` 提供的功能。
- **子节点键名**：统一使用 `children`。可以建立一个基础的树形结构 VO 供其他树形结构继承。
- **基础树 VO**：
  ```json
  {
    "children": [],
    "parentId": "父节点 ID" // 可选，根节点该字段为 null
  }
  ```
- **邻接表**：系统菜单是比较简单的树形结构示例，可以使用邻接表实现。
- **闭包表**：下列情形使用闭包表（Closure Table）实现树形结构：
  - 树的层级较深，且经常需要查询某个节点的所有子孙节点。
  - 需要频繁地进行节点移动操作（如改变节点的父节点）。
  - 需要高效地查询节点的祖先路径。

#### 4.1.4. 查询与操作

- **参数化查询**：所有数据库查询都必须使用参数化（Prepared Statements），以从根本上防止 SQL 注入。
- **避免 `SELECT *`**：查询时必须明确指定需要的字段，减少不必要的数据传输和数据库开销。
- **避免 N+1 查询**：严禁在循环中执行数据库查询。应通过 `IN` 查询或 `JOIN` 将多次查询合并为一次。
- **复杂查询**：对于涉及多表连接（JOIN）、聚合、子查询等复杂场景，推荐使用原生 SQL 或数据库访问库提供的查询构建工具，以保证代码的可读性和性能。

#### 4.1.5. 事务 (Transaction)

- **原子性**：任何涉及多个写操作的业务逻辑都必须封装在事务中，以保证操作的原子性。
- **缩短事务时长**：避免在事务中执行耗时的操作，如外部 API 调用、文件 I/O 等，以减少锁竞争和数据库压力。
- **隔离级别**：默认 `READ COMMITTED`；对一致性要求更高的读-写流程可提升到 `REPEATABLE READ`（评估死锁与性能）。

#### 4.1.6. 索引

- 应当根据查询需求合理创建索引，以提升查询性能。

#### 4.1.7. 数据库结构变更 (Migration)

- **禁用自动同步**：在生产环境中，必须禁用任何自动同步数据库结构的功能。
- **迁移脚本**：所有数据库结构变更（DDL）都必须通过迁移脚本进行管理。每个迁移脚本应是原子的、可回滚的。
- **工具**：Go、Java 等项目统一使用 `Flyway` 进行数据库迁移管理。对于使用了 TypeORM 的项目，简单场景可以用 TypeORM 自带的迁移工具，但复杂场景仍然推荐使用 `Flyway`。
- **命名**：Flyway 命名： `{时间}__V{版本号}__{描述}.sql`，例如 `2025_11_06_15_34__V1__create_user_table.sql`。

### 4.2. 缓存 (Redis)

对于需要使用 Redis 的项目，应遵循以下规范：

- **实例隔离**：每一个项目应使用独立的 Redis 实例，避免多个项目共用同一个实例。也要有独立的 Redis 数据库（DB），避免不同项目的数据混杂在一起。不同环境原则上连接到不同的 Redis 服务器实例，避免开发环境误操作生产环境数据。
- **连接管理**：应使用连接池来管理 Redis 连接，避免频繁创建和销毁连接。
- **键命名**：应该有当前项目名称的前缀，例如 `myproject:user:captcha`。
- **数据序列化**：对于 JSON 等内容，应当序列化成字符串后再存储到 Redis 中，避免使用原生对象存储，防止出现兼容性问题。读取的时候要反序列化。
- **过期时间**：对于缓存数据，必须设置合理的过期时间（TTL），防止缓存雪崩。过期时间应当有抖动，避免大量缓存同时失效。抖动可以使用第三方库，也可以自行封装。
- **数据结构**：根据具体业务场景，选择合适的数据结构（字符串、哈希、列表、集合、有序集合等）来存储数据，避免滥用某一种数据结构。
- **批量操作**：合理使用批量操作，对于大量数据的读写，应使用管道（Pipeline）或 `MGET` 等批量命令。
- **优雅降级**：当 Redis 不可用时，应能优雅降级，不影响核心业务功能。
  - 如果性能允许，可以直接回退到数据库查询。
  - 使用本地缓存（如内存缓存）作为二级缓存。
- **重试机制**：对于网络抖动等临时性故障，应实现合的重试机制（建议最多 3 次，指数退避）。
- **安全**：
  - 生产环境必须为 Redis 设置强密码，并通过环境变量配置。
  - Redis 应部署在内网，不应直接暴露到公网。
- **性能与资源**：
  - 在生产环境中，严禁使用 `KEYS` 命令遍历所有键，应使用 `SCAN` 命令进行增量迭代。
  - 设置 `maxmemory` 限制 Redis 使用的最大内存，防止因内存泄漏导致服务器 OOM。
  - 避免存储过大的键值对（例如，超过 100KB）。对于大对象，应考虑将其拆分为多个小的哈希字段，或将其存储在对象存储中，Redis 只保存其索引。
  - 对于可预见的热点数据（如秒杀商品），应在架构设计上进行优化，例如采用多级缓存或本地缓存，避免所有请求都集中到单个 Redis Key 上。

## 5. 安全与鉴权

### 5.1. 鉴权与权限

- **基础鉴权**：使用 `JWT`，如果项目有必要则引入 `OAuth2.0`。
- **权限模型**：后端项目应当使用基于角色的访问控制（RBAC）模型来管理 API 权限。每个 API 方法都应对应一个唯一的权限点。
- **权限点命名**：权限点的标识符应与 API 的方法名严格对应，以实现精细化的访问控制。例如，`create` 方法对应的权限点为 `create`。
- **查询模式权限**：
  - 查看已删除数据的能力应被视为特殊权限。
  - 当 `mode=withDeleted` 时，用户需具备 `withDeleted` 权限。
  - 当 `mode=onlyDeleted` 时，用户需具备 `onlyDeleted` 权限。
  - `findAll` 由于查出的数据量过于巨大，为了防止被滥用导致性能问题，所以必须要有独立的权限点 `findAll`。
- **软删除权限**：
  - `remove`/`removeMany`（软删除）和 `restore`/`restoreMany`（恢复）操作也必须有独立的权限点。

### 5.2. 安全规范

- **密码存储**：密码必须加盐，使用 `argon2`。
- **输入校验**：虽然客户端会进行输入校验，但后端必须对重要输入进行再校验。
- **敏感数据**：所有敏感数据（如密码、API 密钥等）在存储和传输过程中必须进行加密处理。严禁在日志或 API 响应中直接输出密码、密钥、身份证号等敏感信息。
- **速率限制**：对于登录、注册等敏感接口，应实现速率限制，防止暴力破解攻击。

## 6. 横切关注点

### 6.1. 配置管理

- **环境变量优先**：所有配置项（如数据库凭证、API 密钥、服务端口等）应优先通过环境变量读取。
- **配置文件**：可以提供一个默认的配置文件，但不应将包含敏感信息的配置文件提交到版本控制系统中。
- **环境分离**：应为开发、测试、生产等不同环境提供独立的配置。

### 6.2. 日志规范

- **格式**：结构化 JSON 日志；包含 `level`, `timestamp(UTC)`, `logger`, `message`, `requestId`, `userId`（如有）。
- **级别**：生产记录 `INFO+`，本地可 `DEBUG`。
- **安全**：禁止输出密码、密钥、身份证号等敏感信息。
- **管理**：采样与滚动策略，集中收集（ELK/Cloud Logging）。

### 6.3. 异常处理

- **封装数据库错误**：严禁将原始的数据库错误信息直接暴露给客户端。
- **统一处理**：应使用全局异常处理机制来捕获数据库异常，并将其转换为标准化的错误响应。
- **记录日志**：在捕获异常时，应记录详细的错误日志，以便于问题排查。

## 7. 开发与部署

### 7.1. 文档

- **API 文档**：必须使用 OpenAPI(Swagger) 生成 API 文档。
- **文档内容**：每个接口都应包含清晰的描述、参数说明、响应示例。
- **关键信息**：标注鉴权要求与权限点。为分页/树/复杂过滤提供示例。

### 7.2. 容器化

如果项目使用容器化部署，应遵循以下规范：

- **镜像优化**：使用多阶段构建优化镜像大小。
- **配置**：配置通过环境变量传递，避免在镜像中硬编码配置。敏感配置从密钥管理注入。
- **安全**：非 root 运行；尽量只读文件系统，必要可挂载可写目录。
- **运行时**：
  - **优雅终止**：处理 SIGTERM。
  - **时区同步**：容器内使用 UTC。
